## OpenClash “无订阅信息”的当前根因

当前 UI 点击刷新后，配置文件更新时间可以更新，但订阅信息仍显示“无订阅信息”。截至 2026-07-19 18:46，WAN 已接通，出口路径已闭环；当前根因是远端订阅接口返回 `HTTP 429`，没有返回 `subscription-userinfo` 响应头。

WAN 与公网连通证据：

```text
eth1: LOWER_UP
carrier=1
operstate=up
speed=1000
duplex=full
wan up=true
wan address=192.168.0.30/24
default via 192.168.0.254 dev eth1 proto static src 192.168.0.30
PING_HARD_ROUTER_OK
PING_IP_OK
nslookup em.mesl.cloud 223.5.5.5 -> 172.67.73.69, 104.26.12.155, 104.26.13.155
```

订阅接口脱敏探测证据：

```text
SUB_HOST=em.mesl.cloud
SUB_UA=clash-verge/v2.4.5
CURL_RC=0
HTTP_LINE=HTTP/2 429
CONTENT_TYPE=text/plain; charset=UTF-8
CONTENT_LENGTH=17
HAS_SUBINFO=no

fallback UA=Quantumultx
HTTP_LINE=HTTP/2 429
HAS_SUBINFO=no
```

控制器逻辑证据：

- `sub_info_get` 会按当前配置名 `MESL_家宽` 查订阅 URL。
- 当前运行 YAML 没有 `proxy-providers:` 块，因此控制器回退到 UCI 的 `config_subscribe` 地址。
- UCI 中 `config_subscribe[0].name='MESL_家宽'`，与当前配置文件 basename `MESL_家宽` 匹配。
- `fetch_sub_info()` 只有在远端 HTTP 200 且响应头包含 `subscription-userinfo` 时才会构造 provider 数据。
- 当前远端返回 429 且没有 `subscription-userinfo`，所以 JSON 中 `providers=[]`，页面只能显示“无订阅信息”。

不要把这个现象误判为 UI 时间刷新 bug。当前 UI 时间刷新已由 Edge 黑盒验证：

```text

CLICKED_AT=2026-07-19 17:39:27
AFTER=更新时间: 2026-07-19 17:39:27
截图: D:\Banana_Pi_r3mini\tmp\openclash-refresh-edge-20260719-173924.png
```

## 后续修复路径

先选定拓扑，再写配置。不要同时混用两套出口模型。

### 路径 A：独立软路由拓扑

使用 BPI-R3 Mini 作为下游软路由时：

```text
硬路由 LAN -> BPI WAN eth1
BPI LAN eth0 / BPI Wi-Fi -> 电脑和手机
```

验收条件：

```text
eth1 carrier=1
wan up=true
ip route show 出现 default route
ping 223.5.5.5 成功
订阅域名可解析并可访问
OpenClash UI 刷新后显示当前时间和订阅信息
客户端网关/DNS 指向 BPI
```

截至 2026-07-19 18:46，路径 A 的 WAN 链路、WAN DHCP、默认路由和公网 ping 已通过；未通过项只剩订阅接口响应头。不要高频手动刷新订阅信息，避免继续触发远端 429。等待远端限流窗口恢复后，只点一次刷新并复测 `subscription-userinfo`；如果仍返回 429，改查订阅服务商侧限流、账号状态或订阅接口策略。

### 路径 B：同网段旁路/主路由 LAN 拓扑

继续使用 `192.168.0.242/24` 并只接 `eth0` 时，设备需要明确默认网关和 DNS，否则路由器自身无法更新订阅：

```text
默认网关: 192.168.0.254
上游 DNS: 192.168.0.254 或明确的公共 DNS
```

写入这类配置属于修改 `/etc/config/network` 或相关 DNS 行为。执行前必须：

1. 做当前配置备份。
2. 明确用户选择路径 B。
3. 写入最小变更。
4. 复测默认路由、公网 ping、订阅响应头和 LuCI 截图。

## 可复现只读复核命令

在 Windows 项目根目录执行：

```powershell
python .\openclash-ui-patch\router_readonly.py 192.168.0.242 "date '+%Y-%m-%d %H:%M:%S %z'; ip -4 addr show br-lan; ip route show; /etc/init.d/dnsmasq status; /etc/init.d/openclash status; /etc/init.d/nikki status 2>/dev/null || true; uci -q get network.lan.ipaddr; uci -q get network.@device[0].ports; uci -q get network.wan.device; uci -q get wireless.radio1.channel; uci -q get wireless.radio1.htmode; uci -q get wireless.radio1.disabled; uci -q get r3mini_fan.main.high_up; for f in carrier operstate speed duplex; do [ -r /sys/class/net/eth1/$f ] && printf '%s=' $f && cat /sys/class/net/eth1/$f; done; ping -c 1 -W 2 192.168.0.254 >/dev/null 2>&1 && echo PING_HARD_ROUTER_OK || echo PING_HARD_ROUTER_FAIL; ping -c 1 -W 2 223.5.5.5 >/dev/null 2>&1 && echo PING_IP_OK || echo PING_IP_FAIL"
```

通过判据：

```text
管理地址与用户指定一致
fan high_up=40000
OpenClash running
Nikki inactive
5G channel=48, HE80, enabled
如果要更新订阅，则必须存在 default route 且 PING_IP_OK
```
