# #TEST — 派 subagent 3 段验收

## 必跑条件

新写或修改 skill、项目文档元规约，或增删改命令 / 代码及核心行为后必须运行；不跑即未闭环。

## 黑盒验收

用当前 CLI 的 agent primitive 启动 fresh、无主线会话上下文的 subagent，只提供产出物（Markdown、修改后的代码或命令）和必要基础指引。要求输出：

| 段 | 内容 | 硬要求 |
|---|---|---|
| 阅读理解 | 复述 skill / 命令的目标、关键步骤、失败处置 | 2–4 句 |
| 测试结果 | 按产出物实际执行并粘贴真实输出 | 禁止只复述；无法测试时说明原因 |
| 疑问 | 列出不清楚、矛盾、缺 error case 或缺参数说明之处 | 没有则显式写“无” |

Prompt 必须指定：cwd 为项目根；只 Read 指定文件的绝对路径；用中文和 3 段 Markdown；测试结果实际执行命令 / shell / python 并粘贴真实输出。

## 主线迭代

改产出物，不改 subagent 的理解。出现任一情况就换 fresh subagent 再跑：

- 核心步骤的阅读理解与主线原意不一致。
- 测试失败且产出物未文档化该失败。
- 出现 ≥ 2 条实质性疑问。

疑问 ≤ 1、理解与意图对齐且测试全 PASS 时收敛。第 N+1 轮更换 subagent 类型、prompt 措辞或提问角度。

## 多 CLI 自足性元验证

在 Codex CLI、独立 Claude Code 或任意 fresh 无上下文 subagent 上运行。固定 cwd 为项目根；禁止读取聊天、memory、用户全局；仅允许 workspace 可写沙盒；用户原 prompt 一字不改。

| 步骤 | 动作 |
|---:|---|
| 1 Baseline | 不改文档先跑一次；用 `--json` 记录 tool events |
| 2 诊断 | 检查是否按需 `dd get` 任务相关 doc、显式使用 UTF-8 read/write、避免硬编码列表 / 静态扫源码 / 双份真相源、主动派黑盒 subagent、未触碰 memory / 用户全局 |
| 3 微调 | 只改文档 |
| 4 隔离 code 改动 | 从已知 baseline 建临时副本或 worktree，再启动新 CLI 会话；禁止在工作目录 blanket restore 或删除新增文件 |

| 失败 | 文档修法 |

## Workflow templates

使用当前 CLI 可用的 workflow / parallel-agent primitive；没有专用 Workflow tool 时按相同结构编排。修改 doc 元规约时优先联跑 B + C（K=4）。

| 模板 | 用途 | 成本 |
|---|---|---:|
| A A/B 变体测试 | 产出 / 校准写 doc 规矩，分类 must / optional / harmful | 3N+1 |
| B Harness pass with understand-check | 批量压密度，验证 fact 保留和语义一致 | 3M+K+1 |
| C Understand-check | 探测单 doc 语义清晰度 | K+1 |
| D Fanout-and-review | 卸载主线阅读，多角度分析并合成 digest | N+1；N 推荐 3–5 |

### A：A/B 变体测试

1. 用 fictional 命令 canonical doc V0 隔断 agent 训练记忆；生成 V1..VN，每个只改一个元素：删 params 表、改散文、使用占位示例、加 emoji、只跨引用、删 `error_kind` 列、删失败节或省略输出体。
2. 构造 `pipeline(VARIANTS, answer_stage, judge_stage)`，尾部加 synth。
3. `answer_stage`：每个变体并行派 2 个无上下文 agent，只喂 inline doc 和 6–7 道探针题。
4. `judge_stage`：每变体派 1 个 judge，每题打 0–2 分，统计 `invention_count` 并给 verdict。
5. synth 聚合 must-have / optional / harmful，并保留一句话结论。
6. 禁 agent 使用 Read / Grep / Web / 训练记忆。

此模板只验证 doc 元素必要性，不验证命令实现质量。

### B：Harness pass with understand-check

1. **Analyze**：每 doc 并行派 1 agent，只 Read 目标 doc，输出 `fact_inventory`（命令、路径、阈值、数字、`error_kind`、规则名、证据 marker）、`bloat_candidates`、`low_density_signals`。
2. **Rewrite**：每 doc 派 1 agent，读取原 doc 和 inventory，输出完整新 body；保留全部 fact，故意删除的逐项写入 `intentional_removals`；优先 imperative、bullets，表格更紧凑时优先表格；删除 preamble、welcome、emoji、表面词；folder-level 引用仅用于 inventory / 发现，不得替代本篇必需事实；禁止 file-level 引用；合并相近节；禁止 Markdown 脚注 `[^n]`。
3. **UnderstandCheck**：并行 K 个无上下文 agent（推荐 K=4），只读 new body，回答相同探针题。
4. **Audit**：每 doc 派 1 agent，只读原 doc、inventory、新 body、removals，输出 `retention_pct`、`missing_facts`、`style_violations`、`verdict`。
5. **Synthesize**：聚合 `understanding_consistency_pct`、分类清单和 verdict。

| verdict | retention | consistency | 附加条件 |
|---|---:|---:|---|
| apply | ≥95 | ≥90 | 无严重 style，且行数减少 |
| revise | 80–94 | 70–89 | 或存在小 style 问题 |
| reject | <80 | <70 | 或严重丢失 fact |

此模板只验证事实保留和 doc 语义一致，不验证新 doc 的操作正确性。

### C：Understand-check

并行 K ≥ 3 个无上下文 agent；每个只读 inline doc，禁 Read / Grep / Web，回答同一组 6–7 道探针题；合成 `understanding_consistency_pct` 与 `ambiguous_questions`。它验证清晰度而非正确性；一致度只表示内部一致，所有 agent 可能被同一歧义误导。

### D：Fanout-and-review

通用编排最低 `N ≥ 2`；若具体 tag 规定更高门槛，以具体 tag 为准。

主线不读大批 raw 内容：

1. 并行派 N ≥ 2 个 answer agents，按不同角度或分片读取，并用 schema 强制结构化输出。
2. 派 1 个 review agent，只读结构化 answers，不重读 raw。
3. 主线只读 review 的单一 message。

该模板只定义编排 pattern；机制正确性必须在真实任务跑一次验证。

### Apply 后人工补丁

| 风险 | 处置 |
|---|---|
| rewrite 把 heading 改成反义 | 回滚 |
| `<name>` 被写成 `&lt;name&gt;` | 替换回原字符 |
| audit 漏掉一句话结论 | 补回 |
| rationale / 上下文被当 style 违规整体删除 | 判断是否为关键上下文，必要时恢复 |
| section header 全部 flatten | 需要时恢复 |

若专用 Workflow tool 把 script 存入会话临时目录，跨 CLI 冷启动仍无法读取；按结构即时编排。Templates 是结构契约，不是逐字节脚本。

## 禁止

- 主线自审充当验证，或让 subagent 携带主线上下文。
- 跳过测试结果，只做阅读理解。
- 忽略疑问并直接判定“够用”。
- 一次反馈后即结束，明显偏差却不重跑。
- 把 Understand-check 当机制验证：描述清楚不等于机制正确。
- 把 N-agent 一致度当事实正确：内部一致不等于事实准确。
