# 盘路由目标与拓扑

> 当前状态更正（2026-07-19）：实机管理地址为 `192.168.0.242`，`br-lan=eth0`、WAN=`eth1`。文中的 `192.168.101.0/24` 是独立 LAN 的设计候选，不是当前地址；不得据此访问或写入设备。上游只能接 WAN `eth1`，下游客户端/AP 只能接 LAN `eth0`，两侧不得同接上游交换机。

## 目标

将 Banana Pi BPI-R3 Mini 用作局域网的主软路由，而不是部署完整飞牛 NAS。

- 提供 4G/5G、上游网线或手机热点三种 WAN 上行。
- 让台式机、笔记本和至少一台其他设备统一经软路由出网。
- 在 NVMe SSD 上运行 Docker 与简单共享盘服务。
- 保留电脑自行建立的业务 VPN；软路由只路由和 NAT 加密流量。
- 按需启用已有的代理软件；不得记录订阅链接或凭据。

## 目标拓扑

```text
手机热点 / 5G 流量卡 / 上游网线
                │
      BPI-R3 Mini（主路由）
      ├── Docker 与共享盘（NVMe）
      ├── DHCP、DNS、NAT 与出口策略
      ├── 自身 Wi-Fi（可选接入）
      └── LAN ── 硬路由 LAN（AP 模式）
                       ├── 台式机
                       └── 笔记本与其他 Wi-Fi 客户端
```

硬路由仅承担 AP 和交换机职责：关闭 DHCP，使用软路由 LAN 网段中的固定管理地址，且只使用 LAN 口上联。若软路由 LAN 接入硬路由 WAN 且硬路由继续处于路由模式，会形成双重 NAT，增加端口映射和局域网互访复杂度。

## 网关与 Clash 接管判定

按“客户端是否经过软路由 → 测试连接是否进入 OpenClash”两步判定，分别记录结论。

1. **确认客户端路径。** 当上游网线来自现有硬路由时，LuCI 显示软路由 WAN 的网关和 DNS 为 `192.168.0.254` 属于正常上游状态，不能证明客户端绕行。连接软路由 LAN 或自身 Wi-Fi 的客户端应获得 IPv4 地址 `192.168.101.x`，默认网关和 DNS 均为 `192.168.101.1`。若客户端自身获得 `192.168.0.x` 且默认网关为 `192.168.0.254`，则按本项目目标判定其未经过软路由三层转发；检查 SSID 绑定网络、LAN DHCP、`br-lan` 成员、物理端口和硬路由 DHCP。
2. **确认 OpenClash 接管。** 关闭客户端蜂窝数据和 VPN，启用 OpenClash 后发起一个新的网络连接，同时按客户端 IP 查看实时连接。看到该连接及其匹配规则，只证明该次连接进入 OpenClash；没有记录时检查访问控制、透明代理和防火墙接管。
3. **分别验证双栈。** 分别检查 IPv4 和 IPv6 的地址、默认路由、DNS 和公网出口；IPv4 另按上述租约判定。不得用一种协议或一次连接的结果推断全部流量均已接管。

目标 IPv4 链路：

```text
客户端 192.168.101.x → BPI LAN 192.168.101.1 → OpenClash / NAT → BPI WAN → 上游 → 互联网
```

软路由使用现有硬路由作 WAN 上游会产生双重 NAT，属于过渡接法；若将同一硬路由改作目标拓扑中的下游 AP，则关闭其 DHCP、把管理地址迁入 `192.168.101.0/24`，并改用 LAN-to-LAN 上联。手机当前的 IP、网关、DNS、BSSID、IPv4/IPv6 默认路由及公网出口尚未现场复核，不得记录“已绕过软路由”或“已由 Clash 接管”。

## 存储与 Docker

- 选用不少于 512 GB 的 NVMe SSD。
- 系统继续保留在现有启动介质；NVMe 格式化为 ext4。
- 将 Docker 数据根目录放在 NVMe，例如 `/mnt/data/docker`。
- 将简单共享目录放在 NVMe，例如 `/mnt/data/share`。
- 只运行轻量容器与基础共享服务；不部署完整飞牛、影音转码、Immich 或大型数据库。

安装 SSD 前开盖确认是否存在空闲 NVMe M.2 插槽、插槽 Key 位及其与 RM500CNV 的关系；未确认前不得购买或拆装。当前没有该物理布局的上机证据。

## 手机热点作为 WAN

软路由以无线客户端（STA）连接手机热点，并将该接口作为 `wwan` / WAN 的 DHCP 客户端。笔记本可连接硬路由 Wi-Fi 或软路由 Wi-Fi，流量均经软路由、再由手机热点出网。

优先让手机热点使用 2.4 GHz、软路由或硬路由向客户端提供 5 GHz Wi-Fi，减少同频收发带来的吞吐下降。

## 验收

1. 三台客户端均从软路由获取 IP、网关和 DNS，并可稳定访问互联网。
2. 手机热点作为 WAN 时，客户端流量可经 `wwan` 出口。
3. 电脑建立业务 VPN 后连接正常；不得对 VPN 做解密、代理终止或流量拦截。
4. NVMe 在重启后自动挂载，Docker 数据不落在系统闪存。
5. 至少一个轻量 Docker 容器和一个共享目录可用，且系统没有内存不足或频繁重启。
