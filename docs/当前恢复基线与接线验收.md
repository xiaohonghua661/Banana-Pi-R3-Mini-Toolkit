## 物理接线硬约束

```text
上游主路由 / 上游交换机 ──> eth1 (WAN)
                                    │
                          BPI-R3 Mini
                                    │
PC / 下游交换机 / 下游 AP ────> eth0 (br-lan)
                                    │
                              BPI 自身 Wi-Fi
```

不要把上游主网同时接入 `eth0` 和 `eth1`，也不要把 `eth0` 再接回上游交换机。否则主路由的 DHCP 广播会进入 `br-lan`，无线或下游客户端可能取得上游地址和网关，流量会绕过软路由。

下游硬路由只作为 AP/交换机时，关闭它的 DHCP，使用 LAN-to-LAN 连接到 `eth0`；不要把下游 AP 的 WAN 口用于此连接。

## 改为独立 LAN 前的顺序

1. 先完成配置、软件包、自定义服务和数据的独立备份，并验证归档哈希。
2. 保持 `eth1` 为 WAN、`eth0` 为 LAN；不得通过交换两个设备名来猜测物理口。
3. 先在物理上隔离上游与 `eth0`，再选择一个不与上游重叠的 LAN 网段和 DHCP 池。
4. 应用网络配置后，从 LAN 或 BPI Wi-Fi 客户端检查 IP、默认网关和 DNS 都来自 BPI；没有该证据不得声称流量经过 Clash。
5. WAN 已取得地址和默认路由后，再分别验证 IPv4、IPv6、DNS、OpenClash 连接记录与公网出口。

## 验收与失败处置

| 验收项 | 通过判据 | 失败时处理 |
|---|---|---|
| WAN 链路 | `eth1` 有物理链路，`ifstatus wan` 为 up 且有 IPv4/默认路由 | 保留 LAN 管理，不改无线；检查上游线是否插在 `eth1` |
| LAN DHCP | 客户端取得 BPI LAN 网段的地址、网关和 DNS | 立即停止冲突 DHCP；检查地址池、`br-lan` 和接线，不做端口互换猜测 |
| 双频 Wi-Fi | AP 状态为 ENABLED，客户端可以关联 | 先记录无线状态和信道；不要把扫描权限问题误判为 AP 已关闭 |
| Clash | 新建连接可在 OpenClash 中看到对应客户端/规则 | 先证明客户端网关指向 BPI，再排查透明代理和 DNS |

网络、无线或热点配置需要变更时，先创建新的恢复点；任何验证失败都停止后续写入并保留现场。
